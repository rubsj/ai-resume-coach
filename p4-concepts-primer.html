Project Overview & Why It Matters

P4 is an integration sprint. Unlike P1 (pure generation), P2 (pure evaluation), or P3 (pure fine-tuning), P4 asks you to orchestrate all the patterns you've learned into a single cohesive system: generate data, validate it, analyze failures, correct mistakes, and serve it via an API — all with production-grade engineering.

🎯 The Core Problem

Hiring managers receive hundreds of resumes per job posting. Most are poorly matched. Your system generates realistic resume-job pairs at controlled quality levels, then proves it can identify why a resume fails — not just "it's bad" but specifically: skills gap? seniority mismatch? hallucinated credentials? The difference between a data demo and a production system is that the production system can explain its reasoning.

First Principle

A synthetic data pipeline is only as valuable as its ability to generate controlled variation. Random noise is useless. Controlled degradation — generating "poor fit" resumes with specific, measurable deficiencies — creates training and evaluation data that tests real-world failure modes. This is the same principle behind chaos engineering: you break things intentionally to prove your detection works.

Jaccard Similarity — The Core Metric

Jaccard similarity is the backbone of P4's failure detection. It measures set overlap — given a resume's skills and a job's required skills, how much do they intersect? This is fundamentally different from the cosine similarity you used in P2/P3, which measures vector direction. Jaccard operates on discrete sets, not continuous vectors.

The Math

Jaccard Index Formula

Given two sets A and B:

J(A, B) = |A ∩ B| / |A ∪ B|

Result ranges from 0.0 (no overlap) to 1.0 (identical sets).

The intersection (∩) counts items in both sets. The union (∪) counts items in either set. Division gives you the proportion of shared items.

# WHY Jaccard over cosine: Skills are discrete items, not continuous vectors.
# "Python" either IS or ISN'T in the set — there's no "partially Python."
# Cosine similarity is for comparing vectors in embedding space.
# Jaccard is for comparing sets of categorical items.

def jaccard_similarity(set_a: set[str], set_b: set[str]) -> float:
    """Calculate Jaccard similarity between two skill sets."""
    # WHY guard against empty union: if both sets are empty,
    # |A ∪ B| = 0, which means division by zero.
    # Convention: J(∅, ∅) = 1.0 (two empty sets are "identical")
    if not set_a and not set_b:
        return 1.0

    intersection = set_a & set_b          # Python set intersection operator
    union = set_a | set_b                 # Python set union operator
    return len(intersection) / len(union)

# Example from the spec:
resume_skills = {"python", "javascript", "react", "node"}
job_required  = {"python", "javascript", "docker", "kubernetes"}

# intersection = {"python", "javascript"} → 2 items
# union = {"python", "javascript", "react", "node", "docker", "kubernetes"} → 6 items
# Jaccard = 2/6 = 0.33 → poor overlap

Jaccard vs Cosine — When to Use Which

MetricInput TypeRangeUse Case
Jaccard Discrete sets (skills, tags) 0.0–1.0 P4: "Does this resume have the right skills?"
Cosine Continuous vectors (embeddings) -1.0–1.0 P2/P3: "Is this text semantically similar?"
Dice coefficient Discrete sets 0.0–1.0 Weights intersection 2×: 2|A∩B| / (|A|+|B|). Emphasizes overlap more.
Java / TypeScript Parallel

Jaccard is equivalent to: Sets.intersection(a, b).size() / Sets.union(a, b).size() in Guava. In TypeScript, think of it as comparing two Set<string> objects. The key insight: Python's set operators (& for intersection, | for union) make this a one-liner, whereas in Java you'd need multiple method calls. Python sets are hash-based (like HashSet<String>), so these operations are O(min(|A|, |B|)).

The Fit Level Thresholds

The spec defines 5 fit levels based on skill overlap. Jaccard is the primary signal:

Fit LevelSkill OverlapWhat It Means
Excellent80%+Resume covers nearly all required skills
Good60–80%Strong match, a few gaps
Partial40–60%Significant gaps but relevant experience

Mismatch<20%Wrong domain entirely
Critical Gotcha — Normalization Before Jaccard

Jaccard on raw skill strings will be artificially low. "Python", "Python 3.10", "python developer" are all the same skill but Jaccard treats them as 3 different items. You must normalize skills before computing Jaccard — see Section 4.

🔍 Analogy — The Venn Diagram

Jaccard similarity is literally a Venn diagram as a number. Draw two circles: one for resume skills, one for job skills. The Jaccard score is the area of overlap divided by the total area of both circles combined. A score of 0.8 means the overlap nearly fills both circles. A score of 0.15 means the circles barely touch — fundamentally different skill sets.

Controlled Synthetic Generation

In P1, you generated data and hoped it was realistic. P4 goes further — you generate data with intentional, measurable deficiencies. This is harder than it sounds. Asking an LLM to "generate a poor-fit resume" often produces obviously broken output. The challenge is creating subtle mismatches that test your detection pipeline.

The Generation Matrix

The spec calls for 50 jobs × 5-10 resumes each = 250-500 pairs. Each resume has a target fit level. This means your prompt must control:

What "Controlled" Means

Skill overlap: For a "poor fit" resume against a Python/Docker/K8s job, the resume should have maybe 1 matching skill (say, Python) and several irrelevant ones (say, Photoshop, InDesign).

Experience level: A "mismatch" might be an entry-level candidate for a Staff Engineer role — not just wrong skills, but wrong career stage.

Subtle defects: A "partial fit" should look plausible but have gaps a careful reviewer would catch. This tests whether your labeling system is smarter than a keyword scanner.

First Principle — Adversarial Data Design

The value of synthetic data is proportional to its difficulty. Easy-to-detect failures ("resume is empty") prove nothing. Hard-to-detect failures ("resume has 7/10 required skills but at the wrong seniority level and with 2 hallucinated certifications") prove your pipeline is production-ready. Always generate the hardest test cases you can, because real-world data will be harder still.

Niche Role Detection

The spec requires flagging "niche" roles — positions with unusual or highly specialized skill requirements. Why? Because niche roles generate more failure modes: candidates are less likely to have exact skill matches, and the LLM may hallucinate domain-specific terminology.

04

Skill Normalization — Why Matching Is Hard

This is where many resume-matching systems silently fail. Without normalization, your Jaccard similarity is comparing surface-level strings, not actual skills. "Python", "Python 3.10", "python developer", "python programming" — these are the same skill expressed four different ways.

The Normalization Pipeline

import re

def normalize_skill(skill: str) -> str:
    """Normalize a skill string for consistent matching.

    WHY each step:
    1. Lowercase: "Python" and "python" are the same skill
    2. Strip whitespace: " React " → "react"
    3. Remove version numbers: "Python 3.10" → "python"
    4. Remove role suffixes: "react developer" → "react"
    5. Remove common noise: "js" suffix handling, "framework" stripping
    """
    s = skill.lower().strip()

    # Remove version numbers: "python 3.10" → "python", "node 18.x" → "node"
    # WHY regex: versions appear in many formats (3.10, v2, 18.x, 2024)
    s = re.sub(r'\s*v?\d+[\.\d]*[x]?\s*', '', s).strip()

    # Remove role suffixes: "python developer" → "python"
    # WHY: job listings say "Python developer" but resumes list "Python"
    suffixes = ["developer", "engineer", "programming", "development"]
    for suffix in suffixes:
        if s.endswith(suffix):
            s = s[:-len(suffix)].strip()

    # Handle common aliases
    aliases = {
        "js": "javascript",
        "ts": "typescript",
        "k8s": "kubernetes",
        "postgres": "postgresql",
        "react.js": "react",
        "node.js": "node",
        "vue.js": "vue",
    }
    return aliases.get(s, s)

# Before normalization: Jaccard("Python 3.10", "python") = 0.0
# After normalization: Jaccard("python", "python") = 1.0
The Normalization Trap

Over-normalization destroys signal. If you normalize "machine learning" and "deep learning" to just "learning", you've lost the distinction between two different skills. The alias map should be explicit and tested — don't use fuzzy matching at this stage. You want deterministic, auditable normalization.

Java / TypeScript Parallel

This is equivalent to building a Normalizer<String> function that you'd compose into a pipeline: skill.toLowerCase().trim().replaceAll(versionPattern, "")... In Java you'd use String.replaceAll() with regex; in Python it's re.sub(). The alias map is essentially a Map<String, String> lookup — Python's dict.get(key, default) is the equivalent of map.getOrDefault(key, default).

Why Not Embeddings for Skill Matching?

You could embed skills and use cosine similarity. But for this project, Jaccard on normalized strings is better because: (1) it's deterministic and auditable — you can explain exactly why two skills matched or didn't, (2) it's fast — no model inference, (3) it avoids false positives — "Python" and "JavaScript" have high cosine similarity (both are programming languages) but zero Jaccard overlap. For a resume coach, precision matters more than recall.

05

Hallucination Detection (Rule-Based)

LLMs hallucinate. When generating synthetic resumes, the LLM might produce resumes that claim implausible credentials. Your job is to catch these — not with another LLM (that's the judge, Section 6's territory), but with deterministic rules that are fast and explainable.

The Detection Heuristics

Rule-Based Patterns the Spec Requires

Expert inflation: Entry-level resume (<2 years experience) claiming "Expert" proficiency in 10+ skills. First principles: genuine expertise takes years. An entry-level candidate with 15 "Expert" skills is mathematically implausible.

Skill count explosion: Any resume listing 30+ skills with most marked "Expert." The average developer has 5-8 skills they're genuinely expert in, even after 10+ years.

Magic phrases: "Expert in all," "certified in everything," "proficient in every." These are LLM generation artifacts — no human writes this way.

Timeline impossibilities: Overlapping full-time jobs, getting a senior role after 6 months of experience, graduating college at age 14.

from dataclasses import dataclass
from datetime import date

@dataclass
class HallucinationFlags:
    """Binary flags for each hallucination pattern detected."""
    expert_inflation: bool = False
    skill_count_explosion: bool = False
    magic_phrases: bool = False
    timeline_impossibility: bool = False

def detect_hallucinations(resume: Resume) -> HallucinationFlags:
    flags = HallucinationFlags()

    # WHY threshold of 10: A 2-year developer realistically has 2-4 expert
    # skills. 10+ is a strong signal of LLM hallucination.
    total_years = calculate_total_experience(resume.experience)
    expert_skills = [s for s in resume.skills if s.proficiency == "Expert"]

    if total_years < 2 and len(expert_skills) >= 10:
        flags.expert_inflation = True

    # WHY 30 as threshold: Even senior engineers rarely list 30+ skills.
    # The Dunning-Kruger effect means real experts are more selective.
    if len(resume.skills) > 30 and len(expert_skills) > len(resume.skills) * 0.7:
        flags.skill_count_explosion = True

    return flags

Awkward Language Detection

A related but distinct failure mode: detecting AI-generated or buzzword-heavy text. This isn't about factual correctness — it's about style quality.

# WHY pattern matching over LLM: Fast, deterministic, and the patterns
# are well-defined. You don't need GPT-4 to count buzzwords.

BUZZWORDS = {
    "synergy", "leverage", "paradigm", "disrupt", "pivot",
    "move the needle", "think outside the box", "game changer",
    "best of breed", "low-hanging fruit", "circle back",
    "value-add", "bleeding edge", "deep dive",
}

def detect_awkward_language(text: str, threshold: int = 5) -> bool:
    """Flag text with excessive buzzword density.

    WHY threshold of 5: A summary with 1-2 buzzwords is normal professional
    language. 5+ in a single section signals AI generation or corporate
    buzzword bingo — both red flags for resume quality.
    """
    text_lower = text.lower()
    count = sum(1 for bw in BUZZWORDS if bw in text_lower)
    return count >= threshold
First Principle — Rule-Based vs LLM-Based Detection

Rule-based detection is a floor, not a ceiling. It catches obvious patterns quickly and cheaply. LLM-as-Judge (the optional advanced feature) catches subtle issues that rules miss — but costs money and adds latency. The engineering wisdom: always build the cheap detector first, prove it works, then add the expensive one for marginal gains. This layered approach is exactly how production fraud detection, spam filtering, and content moderation systems work.

LLM Correction Loops — Self-Healing Data

You built a correction loop in P1, but P4's is more sophisticated. Instead of just re-generating a failed record from scratch, P4 feeds the specific validation errors back to the LLM, giving it context about what went wrong. This is a form of closed-loop feedback — the same principle behind PID controllers in engineering and reinforcement learning in ML.

The Correction Flow

Generated Record
      │
      ▼
┌─────────────────┐     ┌──────────────────────┐
│ Pydantic Validate│────→│  VALID → save to      │
│                  │     │  validated_data.jsonl  │
└────────┬────────┘     └──────────────────────┘
         │ INVALID
         ▼
┌─────────────────┐
│ Extract Errors:  │  "email format invalid, end_date before start_date,
│ - field path     │   GPA = 5.2 (must be 0.0-4.0)"
│ - error type     │
│ - invalid value  │
│ - expected format│
└────────┬────────┘
         │
         ▼
┌─────────────────┐     ┌──────────────────────┐
│ LLM Correction   │────→│ Pydantic Re-validate  │
│ Prompt with      │     │                      │
│ error context    │     │  VALID → save         │
└─────────────────┘     │  INVALID → retry      │
                        │  (max 3 attempts)     │
                        └──────────────────────┘

# WHY Instructor handles most of this: Instructor's max_retries parameter
# already implements retry-with-error-context. When Pydantic validation
# fails, Instructor feeds the ValidationError back to the LLM as context
# in the next attempt. This is the same pattern you used in P1.
#
# But P4 adds a NEW layer: post-generation correction for records that
# pass schema validation but fail logical validation (e.g., a "partial fit"
# resume that actually has 90% skill overlap).

def build_correction_prompt(
    original_record: dict,
    errors: list[dict],
) -> str:
    """Build a prompt that gives the LLM specific context about what to fix.

    WHY structured error context: Saying "fix this" is vague.
    Saying "the email field 'john@' is missing a domain, the end_date
    '2020-01-01' is before start_date '2021-06-15'" is actionable.
    The more specific the error context, the higher the correction rate.
    """
    error_descriptions = "\n".join(
        f"- Field '{e['field']}': {e['message']} (got: {e['value']})"
        for e in errors
    )

    return f"""The following resume data failed validation.
Please generate a corrected version that fixes these specific issues:

ERRORS:
{error_descriptions}

ORIGINAL DATA:
{json.dumps(original_record, indent=2)}

Generate a corrected version maintaining all valid fields and fixing only the errors above."""

Tracking Correction Metrics

The spec requires >50% correction success rate within 3 attempts. You need to track:

MetricWhat It ShowsTarget
Correction success ratecorrected_count / invalid_count>50%

🔄 Analogy — The Red Pen Teacher

Imagine a teacher grading essays. The first draft comes back with red marks: "wrong date format here," "missing required section here," "this claim is unrealistic." The student rewrites with those specific corrections. If it still fails after 3 rewrites, the teacher marks it as "needs fundamental rethinking" and moves on. The correction loop is this feedback cycle, automated. The key insight: specific, actionable feedback (not "try harder") is what makes correction effective.

FastAPI — From Pipeline to REST Service

FastAPI is P4's new technology. You haven't used it in P1–P3. It's a modern Python web framework that turns your pipeline functions into HTTP endpoints. If you're coming from Java/TypeScript, FastAPI is like Spring Boot or Express — but with automatic request/response validation via Pydantic (which you already know).

Why FastAPI Specifically

FastAPI's Key Design Decisions

Pydantic-native: Request and response models ARE Pydantic models. Your existing schemas from the generation pipeline become API contracts with zero rewrite. This is the biggest win — no separate DTO layer like in Spring.

Auto-generated OpenAPI docs: Visit /docs and you get an interactive Swagger UI. Free. No configuration. This alone makes it portfolio-worthy — recruiters can test your API without curl.

async-first: Built on Starlette, supports async def endpoints. For P4, this matters because the LLM judge endpoint (<10s) benefits from non-blocking I/O while waiting for the OpenAI API.

Type-driven: Path parameters, query parameters, and request bodies are all typed. FastAPI uses these types for validation, serialization, and documentation simultaneously.

# WHY FastAPI over Flask: Flask requires manual validation, manual docs,
# manual serialization. FastAPI does all three from type hints.
# For a Java developer: FastAPI = Spring Boot + Jackson + Swagger, unified.

from fastapi import FastAPI, HTTPException
from pydantic import BaseModel

app = FastAPI(
    title="Resume Coach API",
    description="AI-powered resume analysis and failure detection",
    version="1.0.0",
)

# Request model — this IS the Pydantic model. No DTO conversion needed.
class ReviewRequest(BaseModel):
    resume: Resume             # Your existing Pydantic Resume model
    job_description: JobDescription  # Your existing Job model
    use_llm_judge: bool = False  # Optional: enable slow but thorough analysis

# Response model — documents EXACTLY what the API returns
class ReviewResponse(BaseModel):
    skills_overlap: float          # Jaccard similarity
    experience_mismatch: bool
    seniority_mismatch: bool
    missing_core_skills: list[str]
    has_hallucinations: bool
    has_awkward_language: bool
    overall_fit_level: str
    recommendations: list[str]

# The endpoint — your pipeline function exposed as HTTP POST
@app.post("/review-resume", response_model=ReviewResponse)
async def review_resume(request: ReviewRequest) -> ReviewResponse:
    """Analyze a resume against a job description.

    FastAPI automatically:
    1. Parses the JSON body into ReviewRequest (via Pydantic)
    2. Returns 422 with error details if validation fails
    3. Serializes ReviewResponse to JSON
    4. Documents this endpoint at /docs
    """
    # Call your existing pipeline functions
    analysis = analyze_resume_job_pair(request.resume, request.job_description)

    if request.use_llm_judge:
        judge_result = await run_llm_judge(request.resume, request.job_description)
        analysis.update(judge_result)

    return ReviewResponse(**analysis)

Correlation Heatmaps & Failure Analysis

Visualization in P4 isn't decoration — it's validation. Your heatmaps prove that your pipeline is working correctly. If "poor fit" resumes don't have more failures than "excellent fit" resumes, something is wrong. If seniority mismatch never co-occurs with skill gaps, that's suspicious.

The 5 Required Visualizations

What Each Chart Proves

1. Failure mode correlation matrix: Which failures co-occur? If seniority mismatch correlates with missing core skills, that's a real pattern — wrong-level candidates usually lack the advanced skills. If all failures are independent, your labeling might be too coarse.

2. Failure rates by fit level: The validation chart. "Excellent fit" should have low failure rates. "Mismatch" should have high failure rates. If this doesn't hold, your controlled generation or your labeling is broken.

3. Failure rates by template: Do certain writing styles cause more failures? "Career changer" templates might trigger more "missing core skills" flags legitimately. "Casual startup" might trigger more "awkward language" flags.

4. Niche vs standard roles: Niche roles should have different patterns — more missing core skills, lower average Jaccard scores.

5. Schema validation heatmap: Which Pydantic fields fail most often? Dates and emails are notoriously hard for LLMs. This informs which validators need strengthening.

import pandas as pd
import seaborn as sns
import matplotlib.pyplot as plt

# WHY seaborn over raw matplotlib: seaborn's heatmap() handles the
# color mapping, annotations, and aesthetics automatically.
# matplotlib is the engine; seaborn is the high-level interface.
# Same relationship as JDBC (low-level) vs Spring Data (high-level).

def plot_failure_correlation(df: pd.DataFrame, output_path: str):
    """Plot correlation matrix of failure modes.

    WHY correlation matrix: Reveals non-obvious relationships.
    If 'hallucinated_skills' and 'expert_inflation' have r=0.85,
    that tells you LLM hallucinations tend to be consistent
    within a single resume — a useful insight for detection.
    """
    failure_cols = [
        "skills_overlap", "experience_mismatch",
        "seniority_mismatch", "missing_core_skills",
        "hallucinated_skills", "awkward_language",
    ]

    # df.corr() computes Pearson correlation between all column pairs
    # WHY Pearson: For binary flags (0/1), Pearson correlation equals
    # the phi coefficient — a standard measure of association for
    # binary variables. No need for a special categorical metric.
    corr_matrix = df[failure_cols].corr()

    fig, ax = plt.subplots(figsize=(10, 8))
    sns.heatmap(
        corr_matrix,
        annot=True,           # Show correlation values in cells
        fmt=".2f",             # 2 decimal places
        cmap="RdBu_r",         # WHY diverging colormap: red=positive, blue=negative
        center=0,              # Center color scale at 0 (no correlation)
        vmin=-1, vmax=1,       # Full correlation range
        square=True,           # Square cells for symmetry
        linewidths=0.5,        # Grid lines between cells
        ax=ax,
    )
    ax.set_title("Failure Mode Correlation Matrix", fontsize=14, pad=20)
    plt.tight_layout()
    fig.savefig(output_path, dpi=150, bbox_inches="tight")
    plt.close(fig)  # WHY close: prevents memory leaks in batch visualization

Java / TypeScript Parallel

JSONL is like Java's BufferedWriter writing one JSON line at a time — you never hold the entire dataset in memory. In Java terms: Files.lines(path).map(JsonParser::parse). JSON is like Jackson's ObjectMapper.writeValueAsString() — the whole object at once. The choice between JSONL and JSON is the same tradeoff as Stream<T> (lazy, line-by-line) vs List<T> (eager, all-in-memory).
